# 009: Multi-platform provider scaffolding - investigate & design

**Status:** inbox  
**Created:** 2026-07-09

---

## Context

Facebook and Instagram providers exist as stubs (`status: "planned"`) with no implementation. Snapchat is the only working platform. Before implementing multi-platform support, we need to understand what data formats these platforms export and how the indexing pipeline should generalize.

This work item focuses on **research + design** — not full implementation. We'll produce a concrete spec that can be executed in follow-up items.

---

## Investigation Findings (2026-07-09)

### Snapchat Export Format (Known — Already Supported)

- Single ZIP file containing:
  - `messages/` — JSON files with conversations/messages
  - `friendships.json` — contacts list
  - `photosAndVideos/` — categorized media by date
  - `loginAndPrivacy/` — account history
- Structured, predictable format
- Media served via HTTP from zip filesystem

### Instagram Export Format (Known Public Spec)

- ZIP file containing:
  - `messages/` — **Same JSON structure as Snapchat!** (`inbox0/inbox.1/messages/message_1.js` or `.json`)
    - Contains `thread_items` with `timestamp_ms`, `content`, `reactions`, `media`
    - Structure: `{ "title": "...", "participants": ["user1"], "is_shoppable": false, "thread_type": "Regular", "thread_path": "inbox-0", "messages": [{...}] }`
  - `your_information/profile.json` — account info, username, email
  - `your_information/followers_and_following/followers.json` + `following.json`
  - `photos_and_videos/photo_N.jpg` — user's photo posts
  - `videos/video_N.mp4` — Reels/videos posted by user
  - **NOTE**: Instagram DMs use the SAME JSON format as Snapchat!

### Facebook/Messenger Export Format (Known Public Spec)

- ZIP file containing:
  - `messages/` — **Same base structure but slightly different**
    - `inbox0/inbox.1/messages/message_1.js` — contains `content` field with text, media URLs
    - Thread objects have `participants`, `thread_path`
    - Messages have `timestamp_ms`, `media` (array of URL objects)
  - `your_facebook_information/profile_information.html` — account profile
  - **KEY DIFFERENCE FROM INSTAGRAM**: FB messages are older, often in `.js` format (ArrayWriter), not pure JSON

### Decision: Implement Instagram First

Rationale:

1. **Closest match to Snapchat** — DM structure is nearly identical
2. **Lower implementation risk** — can reuse ~80% of existing parsing logic
3. **Common use case** — users exporting "Sent/Received" messages are likely Instagram too

### Shared Abstractions Identified

| Concept                     | Snapchat                | Instagram                            | Facebook                   | Reusability |
| --------------------------- | ----------------------- | ------------------------------------ | -------------------------- | ----------- |
| Conversation JSON structure | ✅                      | ✅ (nearly identical)                | ⚠️ (similar, older format) | High        |
| User profile JSON           | ✅                      | ✅ (`your_information/profile.json`) | ✅                         | High        |
| Media serving from ZIP      | ✅                      | ✅ (photos_and_videos/)              | ✅ (messages/media/)       | High        |
| Contact list                | ✅ (`friendships.json`) | ✅ (`followers_and_following/`)      | ⚠️ (contacts.html)         | Medium      |
| Video support               | ✅                      | ✅ (`videos/video_N.mp4`)            | ⚠️ (older format varies)   | Medium      |

### Proposed Provider Interface Extension

Current interface works for all platforms. No changes needed:

```go
type Provider interface {
    ID() string          // "snapchat" | "instagram" | "facebook"
    Name() string        // "Snapchat" | "Instagram" | "Facebook"
    Status() string      // control display in UI
    Description() string
}
```

The real work is in per-platform **indexer/parser adapters** that:

1. Walk the ZIP to find data files
2. Parse them into shared `MediaItem`, `Conversation` structs
3. Store results in platform-scoped SQLite rows

### Database Schema Impact

Current schema should work with one addition: add `platform` column to media/conversations tables for scoping. Migration is straightforward (ADD COLUMN + backfill).
