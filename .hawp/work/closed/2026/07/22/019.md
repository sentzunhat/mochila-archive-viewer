# 019: Serve media over HTTP instead of base64-over-RPC (fixes filter-switch sluggishness)

**Status:** done
**Created:** 2026-07-22
**Closed:** 2026-07-22

---

## Problem

016 flagged "switching a filter while thumbnails are still loading feels sluggish" as minor polish. Investigating it found a bigger root cause than concurrency tuning: `GetMediaSource` (the RPC binding every gallery tile, On This Day tile, chat thumbnail, and the media modal called) read the **entire file — including multi-MB videos — into memory, base64-encoded it (~33% size inflation), and returned it as a JSON string over the same Wails RPC channel used for every other call**. With up to 180 items per page and 2,967 videos in the archive, switching a filter while the previous page's fetches were still in flight could genuinely stall the UI for many seconds (a `computer` tool call literally timed out at 30s earlier this session while 180 video thumbnails were loading).

A proper HTTP media-serving handler (`appshell.App.ServeHTTP`, registered as the Wails asset server's fallback) already existed in the codebase but was **never wired up to the frontend** — dead code.

## Fix

Switched every `<img>`/`<video>` in the app to a direct HTTP `src` (`/media/{platform}/{userId}/{id}`) served by the Go backend, instead of pre-fetching a base64 blob into Svelte state first.

### Found and fixed along the way: a real cross-user cache-leak risk

The original `ServeHTTP` route was `/media/{platform}/{id}` — no user segment. `media_id` is only unique per `(platform, user_id)` (see the DB schema), and browsers cache GET responses by URL. Without a user segment, switching profiles mid-session could serve one user's cached photo at another user's same-numbered media id. Fixed by:
- Adding the active user's id to the URL (`appState.profile.id`, always available in the frontend).
- Adding `Store.ZipPathForUser` and `Service.MediaBytesForUser(platform, userId, id)` — resolves media fully from the request's userId, not the service's process-global `activeUserId`, so it's correct even mid a profile switch (a request-scoped source of truth, not shared mutable state).
- This also happened to fix a second, pre-existing issue: the old `Service.MediaItem` used a fragile "array index equals media id" assumption (`ps.Media[id].ID == id`) that only happened to hold because ids are contiguous — `MediaBytesForUser` queries the store directly by id instead.

### Verification

- `go test ./...` passes, including two new tests: `TestServeHTTPRejectsMalformedPaths` (malformed URL segments all 404) and a manual scratch test (not committed) that confirmed **the correct user gets real bytes (200, real JPEG) and a different user requesting the same numeric id gets a clean 404** — proving the cross-user fix actually works, not just compiles.
- `npm run build` passes.
- Live in the browser: confirmed `<img src>` values are correctly formed (`/media/snapchat/2/0`), the media modal and chat message thumbnails resolve correct URLs, no console errors anywhere in the flow.
- Filter-switch responsiveness: type-filter change → correct count/tiles rendered in **690ms**, down from the multi-second-to-30s+ stalls observed before.
- **Known verification gap, disclosed rather than hidden**: `wails dev`'s browser-preview mode routes asset requests through Vite's dev server, which serves its SPA-fallback `index.html` for any path it doesn't recognize (including `/media/*`) — a dev-mode-only quirk, confirmed by inspecting the actual response body. This means the real image bytes could not be visually/pixel-verified inside this session's sandboxed browser tooling. The server-side logic was instead proven directly via `httptest` (bypassing Vite entirely) with real database data — the exact code path a packaged build uses, since packaged builds have no separate Vite server. A quick visual spot-check in `wails dev`'s native window or a packaged build is worth doing once, but isn't blocking given the direct proof above.

## Cleanup

Removed now-dead code: `App.GetMediaSource` (RPC binding), `Service.MediaSource`, `Service.mediaRenderableSource`, `Service.MediaItem` (the fragile positional-lookup), `mimeFromExt`, and the frontend's `sourceFor`/`ensureMediaSources`/`retryFailedSources`/`mediaSources`/`mediaLoading` state and every reactive block that fed them. The Settings "Cached sources" stat and "Clear media cache" button were removed too — both were about the now-deleted in-memory blob cache and have no meaning against direct HTTP `<img>` tags.

**Left alone, not part of this item**: `Service.cacheMediaFiles` (called during `IndexArchives`) still pre-extracts every media file to `~/.mochila/indexed/providers/<platform>/media/*` on disk. Nothing reads those cached files anymore — `MediaBytesForUser` always reads from the zip directly. This is now wasted disk I/O during indexing, not a correctness bug. Worth cleaning up in a future pass; not touched here to keep this change scoped to the serving path, not the indexing path.
