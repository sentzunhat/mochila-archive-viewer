# 017: Link chat message media to the actual media items (thumbnails in Messages)

**Status:** plan-ready
**Created:** 2026-07-20

---

## Problem (user request, 2026-07-20)

Chat messages with `mediaType: "MEDIA"` currently render as plain text ("Media / attachment") with no visual — even though the referenced photo/video is already indexed in `media_items` and viewable from the Gallery. User asked if these could be linked so chat media shows inline like a real chat client.

## Direct evidence — a viable match exists

`messages.media_ids` is a `" | "`-delimited string (Snapchat's raw JSON field). The **first token** is a Snapchat internal blob id that appears verbatim as a substring inside the corresponding `media_items.entry` filename:

```
messages.media_ids (first token): b~EiASFXNacllMb0NkUlVYaUdweWVVdnFVdTIBVUgCUARgAQ
media_items.entry:                chat_media/2026-06-07_b~EiASFXNacllMb0NkUlVYaUdweWVVdnFVdTIBVUgCUARgAQ.jpg
```

Verified: exact substring match, confirmed against the live `~/.mochila/database.sqlite` (user_id 1, platform snapchat). The largest conversation has 472 media-bearing messages (`alexmaartinez`), so this needs to scale to hundreds of lookups per conversation open, not just a handful.

## Recommended approach (not yet implemented)

Runtime `LIKE '%token%'` scans (no leading-wildcard index use) are workable for one-off lookups but wasteful for hundreds of messages per conversation. Better:

1. **Backend**: when loading a conversation's messages (`Store.loadMessages`), also load all of that platform+user's `media_items.entry` values once, build an in-memory `map[token]mediaID` by extracting the token substring from each entry filename (strip directory, date prefix, and extension), then resolve each message's first `media_ids` token via one map lookup — O(n) total instead of O(messages × media_items).
2. Add a `MediaID *int` field to `types.ChatMessage` (nil when unresolved), populated by this lookup, returned from `GetConversation`.
3. **Frontend**: in the message bubble, when `message.mediaId != null`, render a small thumbnail (reuse `GetMediaSource(platform, mediaId)` + `sourceFor`/`ensureMediaSources` pattern already used in the gallery) that opens the existing media modal on click — reuses `selectedMedia` state, no new modal needed.
4. Cache resolved sources in the existing `mediaSources`/`mediaLoading` maps so switching conversations doesn't re-fetch already-seen media.

## Risks / open questions

- Not every `media_ids` token will resolve (e.g. media from a since-deleted/unindexed zip, or Snapchat's own stickers/bitmoji that aren't in `media_items`) — must degrade gracefully to today's text-only rendering.
- The second `|`-separated segment in some `media_ids` values is a different, longer token (unclear purpose — possibly a duplicate/legacy id) — only the first token was verified to match; treat the rest as unused unless a future audit finds a use.
- Extraction logic (strip date prefix + extension from `entry`) needs to handle the `unknown` date/category edge case already present in the data (1 row observed).

## Acceptance

- [ ] Messages with resolvable media show a thumbnail, not just "Media / attachment"
- [ ] Clicking the thumbnail opens the same media modal used by the Gallery
- [ ] Unresolvable media still renders the current text fallback, no errors
- [ ] Verified live against the `alexmaartinez` conversation (472 media messages) without a visible slowdown opening it
