# 017: Link chat message media to the actual media items (thumbnails in Messages)

**Status:** done
**Created:** 2026-07-20
**Closed:** 2026-07-20 — implemented and verified live

---

## Problem (user request, 2026-07-20)

Chat messages with `mediaType: "MEDIA"` currently render as plain text ("Media / attachment") with no visual — even though the referenced photo/video is already indexed in `media_items` and viewable from the Gallery. User asked if these could be linked so chat media shows inline like a real chat client.

## Direct evidence — a viable match exists

`messages.media_ids` is a `" | "`-delimited string (Snapchat's raw JSON field). The **first token** is a Snapchat internal blob id that appears verbatim as a substring inside the corresponding `media_items.entry` filename:

```
messages.media_ids (first token): b~EiASFXNacllMb0NkUlVYaUdweWVVdnFVdTIBVUgCUARgAQ
media_items.entry:                chat_media/2026-06-07_b~EiASFXNacllMb0NkUlVYaUdweWVVdnFVdTIBVUgCUARgAQ.jpg
```

Verified: exact substring match, confirmed against the live `~/.mochila/database.sqlite` (user_id 1, platform snapchat). The largest conversation has 472 media-bearing messages (`alexmaartinez`), so this needs to scale to hundreds of lookups per conversation open, not just a handful.

## Implementation (2026-07-20)

Built as planned, with one refinement: the token→id map also carries the resolved media item's `type` ("image"/"video"), so the frontend can pick `<img>` vs `<video>` without a second lookup per message (a message's own `mediaType` field, e.g. "MEDIA", doesn't distinguish photo from video).

- **Backend**: `types.ChatMessage` gained `MediaID *int` and `LinkedMediaType string` (both `omitempty`). `Store.mediaTokenIndex(platform, userId)` builds a `map[token]mediaRef{ID, Type}` once per `loadConversations` call (not once per conversation) via `mediaEntryToken()`, which strips the `YYYY-MM-DD_` prefix and extension from `media_items.entry`. `Store.loadMessages` resolves each message's first `media_ids` token through the map. New `Store.MediaItemByID` / `Service.GetMediaItem` / `App.GetMediaItem` binding fetches full metadata for the modal (deliberately not reusing the existing in-memory `Service.MediaItem`, which relies on a fragile array-index-equals-id assumption that doesn't hold for arbitrary ids reached from a message).
- **Frontend**: message bubbles with a resolved `mediaId` render a thumbnail (`<video>` when `linkedMediaType === "video"`, `<img>` otherwise) sharing the existing `mediaSources`/`mediaLoading` cache and `ensureMediaSources` fetch path used by the gallery. Clicking fetches the full `MediaItem` (small client-side cache by id) and opens the same `selectedMedia` modal.
- Added `TestMediaEntryToken` (storage_test.go) — first Go test in the repo — covering the standard dated-prefix case, the no-date fallback, and a `.mp4` entry.

## Live verification

- Conversation "alexmaartinez" (103 messages, 103 media references): 59 thumbnails resolved (53 images, 6 videos), rest correctly fell back to no-thumbnail (unresolvable — likely non-`media_items` attachments like shared links/stickers). Clicking an image thumbnail opened the modal with correct metadata (date, type, category, zip, path). A video message correctly rendered as `<video>` with a `data:video/mp4` source, not a broken `<img>`.
- Largest conversation (1,143 messages, conversation_id `alexmaartinez` — a different one from the 103-message conversation above; two conversations share the display title): opened in 586ms via direct binding timing, resolving 379 of 1,143 messages to media — no visible slowdown.
- `go build`, `go vet` (no new warnings), `go test ./...`, and `npm run build` all pass.

## Risks / open questions (unchanged from plan, still relevant)

- Resolution rate is well under 100% (~57% and ~33% in the two conversations tested) — expected, not a bug: many `media_ids` likely reference shared links, stickers, or media outside the indexed export.
- The second `|`-separated segment in some `media_ids` values is a different, longer token — unused, only the first token was verified to match.

## Acceptance

- [x] Clicking the thumbnail opens the same media modal used by the Gallery
- [x] Unresolvable media still renders the current text fallback, no errors
