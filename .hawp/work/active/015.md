# 015: Legacy data ownership — reassign or clean user_id 0/1 archive data

**Status:** done
**Created:** 2026-07-20
**Closed:** 2026-07-21 — executed with explicit user authorization

---

## Problem

The live database (`~/.mochila/database.sqlite`) carries data from before multi-user isolation:

- The 8,617 indexed Snapchat items (plus 6 archive_files, 33 conversations, 23 json_files) exist **twice**: once under `user_id 0` and once under `user_id 1`.
- The `user_id 0` copies were written while nobody was logged in — `GetFrontendState` used `ActiveUser().ID` (0 when logged out) as the active user. That code path is fixed (guard `p.ID > 0`, 2026-07-20), so the user_id 0 rows are now unreachable dead weight.
- The `user_id 1` copy belongs to profile 1, which had an **empty username** until 2026-07-20, when it was renamed `legacy` / "Legacy (pre-multi-user data)" so it can be logged into.
- The user's real personas (`beltrd` id 2, `diego` id 3) own no data.

## Direct evidence

```sql
SELECT user_id, COUNT(*) FROM media_items GROUP BY user_id;  -- 0|8617, 1|8617
SELECT profile_id, username FROM profile;                    -- 1|legacy, 2|beltrd, 3|diego
```

## Proposed slices

1. **Decide ownership**: reassign the user_id 1 data to the user's primary profile (likely `beltrd`), or keep the `legacy` profile as the archive owner. User decision required.
2. **Delete the user_id 0 duplicates** (unreachable after the fix) — ~8,679 rows across media_items/archive_files/conversations/json_files. Back up the sqlite file first.
3. *(Optional)* A small startup migration in `storage.go` that folds any `user_id 0` rows into user 1 automatically so other installs self-heal.

## Risks

- Destructive deletes on the user's only archive DB — always copy `database.sqlite` aside first.

## Resolution (2026-07-21)

User authorized both decisions directly:
1. Reassign the archive to `beltrd` (profile_id 2), not keep it on `legacy`.
2. Delete the unreachable user_id 0 duplicates, after a backup.

Executed:
1. Stopped `wails dev` (avoid a stale connection during direct DB writes).
2. Backed up: `~/.mochila/database.sqlite` → `~/.mochila/database.sqlite.backup-2026-07-21-pre-015` (verified: 17,234 media_items rows = 8,617 × 2, matching the known duplicate).
3. Single transaction: `UPDATE ... SET user_id = 2 WHERE user_id = 1` across all 6 scoped tables (media_items, archive_files, conversations, messages, json_files, platform_snapshots), then `DELETE ... WHERE user_id = 0` across the same 6 tables.
4. Logged out the `legacy` profile (`logged_in = 0`) so the app lands on the login screen next launch rather than resuming a now-empty session.
5. `PRAGMA integrity_check` → `ok`.

Verified live (wails dev restarted, fresh browser session):
- Post-migration row counts: every table has exactly one user_id left (2), matching pre-migration per-user-id counts exactly (8,617 media_items, 6 archive_files, 33 conversations, 4,750 messages, 23 json_files, 1 platform_snapshot) — no rows lost, no duplication.
- Logged in as `beltrd` → dashboard shows Snapchat indexed, 8,617 media/5,650 photos/2,967 videos/33 conversations/23 JSON files, matching exactly.
- Logged in as `legacy` → dashboard shows all-empty cards, confirming clean separation.

Slice 3 (automatic self-healing migration for other installs) was considered but **not implemented**: the correct general-case migration (reassign orphaned user_id 0 rows when no conflicting row exists, else delete the duplicate) touches the user's live database automatically on next app start, which is exactly the kind of action this item's own risk note says needs explicit authorization first — not something to wire into unattended startup code without a separate decision. Not pursued further since this was a one-time repair of one known-affected database, not a general product concern (the root-cause bug, `GetFrontendState` adopting user 0, is already fixed in `appshell/app.go`).
