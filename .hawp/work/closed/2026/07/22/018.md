- Raw `archive.CheckForUpdate("0.0.1")` correctly returned `{Available:true, Latest:"v0.0.2", URL:"https://github.com/sentzunhat/mochila-archive-viewer/releases/tag/v0.0.2"}`.

013's slice 3 (update check) is now fully verified working end to end, not just unit-tested.

## Problem

Cut `v0.0.1` and `v0.0.2` tags to test the auto-update check built in 013 slice 3. The release pipeline worked and produced real GitHub Releases with downloadable assets — but `CheckForUpdate` never detects the newer version, because **the repository is private** (`gh repo view` confirms `isPrivate: true`), and GitHub's REST API returns `404` for `GET /repos/{owner}/{repo}/releases/latest` on private repos when called without authentication — which is exactly what the shipped app does (and correctly, by design: no credentials are bundled into a distributed binary).

This isn't a bug in `archive.CheckForUpdate` — it degrades exactly as designed (silently, no crash, `Available: false`). It's a design gap: the plan assumed the public GitHub API would be reachable, and it verified that against `go build`/`go test`/local browser testing, none of which would have caught a repo-visibility issue. Confirmed directly:

```
$ gh repo view sentzunhat/mochila-archive-viewer --json isPrivate
{"isPrivate":true}

$ curl -s -o /dev/null -w '%{http_code}' https://api.github.com/repos/sentzunhat/mochila-archive-viewer/releases/latest
404
```

(the 404 was reproduced even after `v0.0.2`'s release was published/un-drafted — ruling out "it just needs to not be a draft" as the fix.)

## Options (user decision needed — not made unilaterally)

1. **Make the repository public.** Matches 013's own original framing ("The project is going open source, so releases must be reproducible from CI on GitHub") — this was very likely always the intended end state, just not yet done. Before flipping visibility: skim the commit history and current tree for anything not meant to be public (API keys, personal file paths beyond what's already in this transcript, etc.) — none found during this session's work, but that's not a substitute for the repo owner's own check.
2. **Keep the repo private; ship an authenticated check.** Requires embedding *some* credential in every distributed binary just to read release metadata — a fine-grained PAT scoped to `metadata: read` only is the least-bad version of this, but it's still a real credential shipped to every user's machine, discoverable by anyone who inspects the binary. Not recommended.
3. **Keep the repo private; publish releases somewhere public instead** (a separate public "releases" repo, a static `version.json` on GitHub Pages, a plain S3/CDN URL, etc.) — decouples "is the source public" from "can the app check for updates." More infrastructure to stand up and maintain than option 1.
4. **Accept that the check silently never finds anything, for now.** Zero risk, zero work — the feature already fails exactly as gracefully as designed. Revisit whenever the repo's visibility question gets settled anyway.

No code changes are proposed until this is decided — implementing option 2 or 3 speculatively would be wasted work if the repo goes public anyway (option 1), and option 1 requires nothing from this codebase at all.

## Secondary finding: darwin-amd64 (Intel Mac) CI leg is stuck

Both `v0.0.1` and `v0.0.2` release builds show `darwin-arm64` and `windows-amd64` completing in 2-3 minutes each, with real assets uploaded — but the `darwin-amd64` (`macos-13` runner) job was still queued (not even started) after **~3 hours** for both tags. This reads as a GitHub-side shared-runner capacity/availability issue for the `macos-13` label specifically (Apple Silicon has been the default for years; Intel runner pools are known to be smaller and increasingly deprioritized), not a problem with `release.yml` itself — the two other legs prove the workflow logic is correct.

Current state: both releases are live and usable with 2 of 3 planned platform assets (macOS Apple Silicon + Windows); the Intel Mac zip is simply missing from each. Not blocking anything — flagging for a future decision on whether Intel Mac support is worth keeping in the matrix given the queue cost, or whether to just accept occasional multi-hour delays on that one leg.
