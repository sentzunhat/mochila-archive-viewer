# 008: Fix media source loading gaps across views and filters

**Status:** inbox  
**Created:** 2026-07-09

---

## Context

When switching views (Gallery → On This Day → Messages) or applying filters, media tiles show "Loading image…" / "Loading video…" indefinitely because `ensureMediaSources()` was only called once on initial page load with a 48-item cap. Items not in that initial batch never get their source paths resolved.

## Root Cause Analysis

### Bug #1: `loadPlatform` caps fetch at 48 items

```ts
await ensureMediaSources(media.slice(0, 48)); // Line 235
```

Only the first 48 media items get source URLs. Everything else sits on "Loading…" forever.

### Bug #2: No re-fetch on view change / filter changes

- `selectedYear`, `selectedCategory`, `selectedType` are reactive ($:) but they only recompute `filteredMedia` — they don't trigger source fetching for newly visible items
- Switching to "On This Day" (`onThisDayMedia`) fetches 0 sources because today's items were never in the initial `media.slice(0, 48)` batch

### Bug #3: "Load more" re-fetches already-loaded items (no-op)

```ts
async function loadMore() {
  visibleLimit += 180;
  await ensureMediaSources(visibleMedia); // visibleMedia includes already-resolved items
}
```

`ensureMediaSources` is correct to skip resolved items, but the real issue is it's called with a slice(0, 96) inside anyway — limiting new fetches to only 96 per call.

### Bug #4: `onThisDayMedia` items never batch-fetched

The Today view renders media items that are computed from `media.filter(...)` — but no code ever calls `ensureMediaSources(onThisDayMedia)`.

---

## Scope

### Task 8.1: Remove the 48-item cap on initial fetch

- **File:** `src/frontend/src/App.svelte`
- **Change:** Fetch sources for all visible items on mount, not just first 48

### Task 8.2: Trigger source fetch on view/filter changes (not full re-fetch)

- Instead of fetching everything, only fetch sources for items that don't have them yet
- Use reactive computed values to detect when new media becomes visible
- Specifically ensure `onThisDayMedia` gets sourced when switching to Today view

### Task 8.3: Cap concurrent batch size to avoid Wails RPC overload

- Keep Promise.all() but cap the actual concurrent calls to ~20 instead of 96
- Adds perceived progress — not everything hangs at once for 5 seconds

### Task 8.4: Add visible-only fetch trigger in onMount lifecycle

- After initial mount + loadPlatform, kick off a second pass that fills remaining visible items
- Handles the case where `visibleLimit` starts at 180 but only 48 were sourced

---

## Implementation Strategy (Reactive)

Instead of imperative calls to `ensureMediaSources`, use a reactive derived value:

```ts
// After view/filter changes, ensure sources for newly visible items
$: if (visibleMedia.length > 0 && !loading) {
  // Only fetch sources for items that don't have them yet
  const unsourced = visibleMedia.filter(
    (m) => !mediaSources[m.id] && mediaLoading[m.id] !== "resolved",
  );
  if (unsourced.length > 0 && unsourced.length <= BATCH_SIZE) {
    ensureMediaSources(unsourced);
  }
}
```

This approach:

- Automatically triggers when `visibleMedia` changes (filter, pagination, view)
- Only fetches items that don't already have sources
- Doesn't loop infinitely because resolved items are excluded

---

## Risks

- Reactive auto-fetch could trigger multiple times for the same batch — guard against duplicate pending requests using `mediaLoading[item.id] === 'pending'` check inside ensureMediaSources
- Large galleries with 10k+ items: reactive re-filters on every keystroke might fire too often — add debounce on search input changes

## Verification

- Open app → Gallery loads all 48 initial images (not stuck on "Loading…")
- Switch to "On This Day" → Today's media tiles show images, not loading placeholders
- Apply category/type filters → Filtered results load their sources correctly
- "Load more" button works without hanging
- Switching between Gallery and OnThisDay multiple times doesn't lose images
