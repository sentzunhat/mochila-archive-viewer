## Context

Full audit of the mochila-archive-viewer Go+Svelte codebase against:
- **Zacatl layered architecture**: Application → Domain → Infrastructure, strict inward dependencies, no layer-skipping
- **SQL safety and consistency**: parameterized queries, correct PK design, error propagation
- **Object/type modeling**: domain purity, no duplication, JSON tag correctness
- **Error handling**: no silent swallowing, correct HTTP status codes
- **Documentation**: package-level GoDoc, exported method comments
- **Dead code**: anything written but never read in the serving path

Sources audited: `src/internal/types/shared.go`, `src/internal/archive/storage.go`, `src/internal/archive/service.go`, `src/appshell/app.go`, `src/frontend/src/App.svelte`, `src/main.go`.

Zacatl docs reviewed: `zacatl/docs/guidelines/framework-overview.md`, `zacatl/docs/guidelines/code-style.md`.

---

## Audit Summary

The codebase has a **sound layered architecture** and correct SQL injection hygiene. The main issues are:
1. A **real data bug** in `platform_snapshots` PRIMARY KEY (multi-user data loss)
2. **Dead code** from the base64→HTTP media migration (cacheMediaFiles, LocalPath)
3. **Silent error swallowing** in `PlatformStats` and `migrate()`
4. **Layer boundary violations**: service directly accesses `store.db`, global `appSettings` var
5. **Documentation gaps**: no package-level GoDoc, most exported Store methods uncommented
6. **Frontend decomposition**: 1369-line App.svelte should be split into components

## Part A — SQL / Storage Layer Fixes (HIGH PRIORITY)

### A1 — `platform_snapshots` PRIMARY KEY is missing `user_id` — data loss bug

**File**: `src/internal/archive/storage.go:267`

**Finding**: The table is defined as `platform TEXT PRIMARY KEY`. The `user_id` column was added later as a backwards-compat `ALTER TABLE ADD COLUMN` but is NOT in the PRIMARY KEY. The `SaveSnapshot` INSERT uses `ON CONFLICT(platform) DO UPDATE` — meaning any second user indexing their archive for the same platform (e.g. "snapchat") silently overwrites the first user's `media_count` and `zip_count`.

**Impact**: Multi-user dashboards show wrong numbers. If users "legacy" and "beltrd" both have Snapchat archives, the dashboard stats will reflect whichever user indexed last.

**Fix**: Migrate `platform_snapshots` to `PRIMARY KEY(platform, user_id)`. Requires a migration step (rename → recreate → copy → rename back) like the profile migration. Update the ON CONFLICT clause to `ON CONFLICT(platform, user_id)`. This is analogous to every other table's compound PK.

**Effort**: Medium — migration code pattern already exists in `migrate()`.

---

### A2 — `loadGroupedCounts` uses string interpolation for a column name

**File**: `src/internal/archive/storage.go:483`

```go
rows, err := s.db.Query(`SELECT `+field+`, COUNT(*) FROM media_items WHERE platform = ? AND user_id = ? GROUP BY `+field, ...)
```

**Finding**: The `field` value is whitelisted (only `"year"`, `"type"`, `"category"` are accepted), so there is no SQL injection risk. However, the pattern will trigger automated security scanners and is inconsistent with the fully-parameterized style used everywhere else.

**Fix**: Replace with three explicit query constants, one per field, and a switch statement. Remove the dynamic string construction.

```go
const groupByYear = `SELECT year, COUNT(*) FROM media_items WHERE platform = ? AND user_id = ? GROUP BY year`
const groupByType = `SELECT type, COUNT(*) FROM media_items WHERE platform = ? AND user_id = ? GROUP BY type`
const groupByCategory = `SELECT category, COUNT(*) FROM media_items WHERE platform = ? AND user_id = ? GROUP BY category`
```

**Effort**: Low.

---

### A3 — `MediaPaginated` uses `fmt.Sprintf` for LIMIT/OFFSET instead of parameterized placeholders

**File**: `src/internal/archive/storage.go:599`

```go
query += fmt.Sprintf(" ORDER BY media_id LIMIT %d OFFSET %d", limit, offset)
```

**Finding**: These are typed `int64` values so there's no injection risk. But SQLite supports `?` placeholders for LIMIT and OFFSET. Using `fmt.Sprintf` is inconsistent with the fully-parameterized pattern used for all other values.

**Fix**:
```go
query += " ORDER BY media_id LIMIT ? OFFSET ?"
args = append(args, limit, offset)
```

**Effort**: Trivial.

---

### A4 — `migrate()` silently ignores errors in profile DDL migration steps

**File**: `src/internal/archive/storage.go:391-394`

```go
s.db.Exec(`INSERT OR IGNORE INTO profile_new ...`)  // error discarded
s.db.Exec(`DROP TABLE IF EXISTS profile`)            // error discarded
s.db.Exec(`ALTER TABLE profile_new RENAME TO profile`) // error discarded
```

**Finding**: If any of these fail (e.g. disk full, permissions), the migrate function returns nil (success) with a broken schema. The `CREATE TABLE profile_new` above correctly checks its error, making the inconsistency even more obvious.

**Fix**: Capture and return errors for all three exec calls.

**Effort**: Trivial.

---

### A5 — `PlatformStats` in `storage.go` silently swallows scan errors

**File**: `src/internal/archive/storage.go:645-653`

```go
s.db.QueryRow(`...image count...`).Scan(&imageCount)  // errors ignored
s.db.QueryRow(`...video count...`).Scan(&videoCount)  // errors ignored
s.db.QueryRow(`...archive_files...`).Scan(&zipCount)  // errors ignored
...
```

**Finding**: The first query (`mediaCount`) correctly returns an error. All subsequent queries silently fall back to zero on any failure. If the DB is unavailable mid-request, the dashboard shows false zeros for image/video/zip/conversation counts.

**Fix**: Propagate errors from all scan calls. Accept that this function either returns all counts or none.

**Effort**: Low.

## Part B — Dead Code from base64→HTTP Migration (MEDIUM PRIORITY)

### B1 — `cacheMediaFiles` and `mediaCachePath` are dead code

**File**: `src/internal/archive/service.go:602-632`

**Finding**: `cacheMediaFiles` extracts all media files from zips to `~/.mochila/indexed/providers/<platform>/media/` during `IndexArchives`. Since the HTTP media handler (`ServeHTTP`) now reads directly from zips via `Service.MediaBytesForUser` → `snapchat.ReadEntry`, the extracted files are **written but never read**. They waste disk space (one copy of every media file on-disk) and slow down every `IndexArchives` call.

Additionally, `types.MediaItem.LocalPath` is set by `cacheMediaFiles` and stored in the DB's `local_path` column, but is never read anywhere in the serving path.

**Fix**:
1. Remove `cacheMediaFiles` and `mediaCachePath` from `service.go`.
2. Remove the `LocalPath` field from `types.MediaItem` or tag it `json:"-"` and stop populating it.
3. Remove `local_path` from DB INSERT and SELECT queries in `storage.go` (or keep the column but stop using it — migration to remove a column requires table recreation in SQLite).
4. Update `IndexArchives` to not call `cacheMediaFiles`.
5. Document in `README.md` that `~/.mochila/indexed/providers/` is now vestigial and can be deleted by users who indexed before v0.1.

**Note**: `ProviderMediaRoot` and related path helpers become unused after this change.

**Effort**: Medium — touches service, storage, types, and README.

---

### B2 — `GetMedia` has an in-memory fast-path that is only reachable right after `IndexArchives`

**File**: `src/internal/archive/service.go:232-253`

**Finding**: When `ps.Index != nil`, `GetMedia` filters the in-memory `ps.Index.Media` slice directly. This fast path is only active immediately after `IndexArchives` completes in the same process lifetime — after a restart, `ps.Index` is nil and the DB path runs. The in-memory path is never exercised in normal usage (user opens app, existing archive loads from DB).

The same dual-path pattern exists in `GetConversations` and `JSONFiles`.

**Fix options**:
- **Simple**: Remove the in-memory paths entirely. Always go through the DB. `IndexArchives` already writes everything to the DB before returning — there's no correctness reason to serve from the in-memory index.
- **Conservative**: Document the dual-path behavior explicitly in each function.

The simple fix is preferred: it eliminates a dead code path, reduces cognitive load, and makes behavior consistent across process restarts.

**Effort**: Low–Medium (removing 3 dual-path branches).

---

### B3 — `DebugJSON` method on `Store` is unexported and unused

**File**: `src/internal/archive/storage.go:936`

**Finding**: `DebugJSON` is an exported method on `Store` that dumps the full snapshot as JSON. It is not called from `service.go`, `app.go`, or any test. It exists as a debug helper from an earlier development phase.

**Fix**: Remove it, or if desired for devtools, expose it through a `Service` method and a Wails binding behind a dev-mode flag.

**Effort**: Trivial.

---

### B4 — `SetArchiveZipsDirect` is exposed as a Wails binding in production builds

**File**: `src/appshell/app.go:157`

**Finding**: `SetArchiveZipsDirect` lets any caller set archive zip paths directly as a comma-delimited string, bypassing the native file picker. It was added to work around native dialog hangs in dev mode. It is exported and registered as a Wails binding — any user of the packaged app can call `window.go.appshell.App.SetArchiveZipsDirect(...)` from the browser devtools to load arbitrary paths.

This is a low-severity issue for a local-only desktop app (the attacker would need local file system access to do harm anyway), but it's a surface area that violates the principle of least authority.

**Fix**: Add a `// go:build dev` build tag to this method, or hide it from the Wails binding by not including it in the `Bind` slice in `main.go`. The file picker binding (`SelectArchiveZips`) is the only user-facing path.

**Effort**: Low.

## Part C — Layer Boundary Violations (MEDIUM PRIORITY)

### C1 — `Service.GetPlatformStats` directly accesses `s.store.db`

**File**: `src/internal/archive/service.go:273`

```go
rows, err := s.store.db.Query(`
    SELECT COUNT(*) FROM media_items WHERE platform=? AND user_id=? AND type='image'
    UNION ALL SELECT COUNT(*) FROM media_items WHERE platform=? AND user_id=? AND type='video'
`, platform, s.activeUserId, platform, s.activeUserId)
```

**Finding**: `Service` is the domain layer; `Store.db` is the infrastructure layer's private field. Service should only call methods on `Store`, never reach through to the underlying `*sql.DB`. This violates the ports-and-adapters boundary.

Additionally, `Store.PlatformStats` already queries `imageCount` and `videoCount` (lines 645-646 in storage.go) — the service is running the same queries a second time and overwriting the store's answers with a UNION ALL variant. This means:
1. The DB is hit twice for image/video counts per `GetPlatformStats` call.
2. The service's UNION ALL approach gives the same result as the store's two individual queries.

**Fix**: Remove the direct `s.store.db.Query` block from `service.go`. `Store.PlatformStats` already returns `ImageCount` and `VideoCount` — the service just needs to trust and return what the store computed.

**Effort**: Low — delete ~20 lines from service.go.

---

### C2 — `appSettings` is a process-global mutable variable

**File**: `src/appshell/app.go:220`

```go
var appSettings = &AppSettings{Pagesize: 180, ProfileID: 1, LoggedIn: false, UpdateCheckEnabled: true}
```

**Finding**: `appSettings` is a package-level singleton pointer that is mutated by both `GetAppSettings` and `SaveAppSettings`. Wails runs handlers in goroutines — concurrent settings reads/writes are not protected by a mutex. For a single-user desktop app the risk is low, but it conflicts with Go's "share memory by communicating" idiom and would be flagged by the race detector under concurrent load.

Additionally, `Homedir` on `AppSettings` is set by `appSettings.Homedir = appSettings.Homedir` — a no-op self-assignment that go vet flags. The field is never populated with a real value.

**Fix**:
1. Move `appSettings` to a field on `App`: `settings AppSettings` (a value, not a pointer).
2. Add a `sync.RWMutex` if concurrent access is possible, or document that Wails serializes handler calls.
3. Remove the `Homedir` no-op line. If the homedir is needed, populate it at startup from `os.UserHomeDir()` in `NewApp` or `Startup`.
4. Add `json:"homedir"` tag to the `Homedir` field so JSON serialization is consistent with the other fields.

**Effort**: Low–Medium.

## Part D — Type Modeling Improvements (LOW-MEDIUM PRIORITY)

### D1 — `Profile` and `UserEntry` are overlapping types

**File**: `src/internal/archive/storage.go:28` and `:930`

**Finding**: `Profile{ID, Username, FullName, LoggedIn}` and `UserEntry{ID, Username, FullName}` differ only in `LoggedIn`. `AvailableUsers()` returns `[]UserEntry` instead of `[]Profile` with `LoggedIn: false/true`. This forces the frontend to handle two types for what is fundamentally the same domain entity.

**Fix**: Remove `UserEntry`. Change `AvailableUsers` to return `[]Profile` (with `LoggedIn` populated from the DB). Update `archive.UserEntry` references in `app.go` and the frontend accordingly.

**Effort**: Low.

---

### D2 — `PlatformSnapshot` in `appshell` duplicates `archive.Snapshot`

**File**: `src/appshell/app.go:54`

**Finding**: `appshell.PlatformSnapshot` and `archive.Snapshot` have nearly identical fields. The appshell struct exists to avoid exposing `archive.Snapshot` directly as a Wails binding type. This is valid separation but is undocumented.

**Fix**: Add a comment explaining the intentional duplication: the appshell struct is the public API contract for the Wails JS bindings and is kept separate from the storage struct to allow each to evolve independently.

**Effort**: Trivial.

---

### D3 — `AppSettings.Homedir` has no JSON tag and contains a no-op self-assignment

**File**: `src/appshell/app.go:214, 227`

```go
type AppSettings struct {
    Pagesize int    `json:"pageSize"`
    Homedir  string          // no json tag → serializes as "Homedir"
    ...
}
// GetAppSettings line 227:
appSettings.Homedir = appSettings.Homedir  // go vet: self-assignment
```

**Finding**: Missing JSON tag means the field serializes as `"Homedir"` (capital H), inconsistent with all other fields. The self-assignment is a go vet warning. The field is never populated with a real homedir value.

**Fix**: Add `json:"homedir"` tag. Remove the self-assignment. If the homedir is needed (e.g. to show the user their data directory), populate it in `GetFrontendState` or `GetAppSettings` from `s.service.StorePath()` which already returns the path. If it's not needed, remove the field.

**Effort**: Trivial.

## Part E — Documentation Gaps (MEDIUM PRIORITY)

### E1 — No package-level GoDoc comments

**Files**: `src/internal/types/shared.go`, `src/internal/archive/storage.go`, `src/internal/archive/service.go`, `src/appshell/app.go`

**Finding**: Go convention is `// Package <name> <description>.` as the first line of a file (or a `doc.go` file) for every package. None of the packages have this. `go doc` and godoc tooling generates empty package-level documentation.

**Fix**: Add package doc comments to each package:
- `types`: "Package types defines shared domain value types for the mochila archive (media items, conversations, messages, index metadata)."
- `archive`: "Package archive provides the storage (SQLite) and service layers for indexing and querying social media export archives."
- `appshell`: "Package appshell exposes the archive service to the Wails frontend via bound methods and HTTP media serving."

**Effort**: Trivial.

---

### E2 — Most exported Store methods lack GoDoc comments

**File**: `src/internal/archive/storage.go`

Uncommented exports: `OpenStore`, `Close`, `Path`, `RootDir`, `ProvidersRoot`, `ProviderRoot`, `ProviderMediaRoot`, `ProviderSnapshotPath`, `SaveSnapshot`, `LoadSnapshot`, `SelectedArchives`, `Summary`, `Media`, `Conversations`, `JSONFiles`, `Conversation`, `PlatformStats`, `WritePlatformSnapshotFile`, `LoadProfile`, `GetProfileByID`, `SaveProfile`, `Logout`, `ActiveUser`, `AvailableUsers`.

Well-commented: `ZipPathForUser`, `MediaItemByID`, `MediaPaginated`'s comment is present but says "filtered by year" instead of "filtered by MediaFilter". `MediaCount` has a comment.

**Fix**: Add single-line GoDoc to each exported function. At minimum document the semantics of `nil` returns (e.g. `LoadSnapshot` returns `nil, nil` for a new user; `Conversation` returns `nil, nil` when not found). Update `MediaPaginated`'s stale comment.

**Effort**: Low (30–40 lines of comments, no code changes).

---

### E3 — `GetMedia` dual-path behavior is undocumented and surprising

**File**: `src/internal/archive/service.go:232`

**Finding**: The function serves from in-memory state right after indexing, then from the DB after restart. There is no comment explaining this. A reader seeing the conditional on `ps.Index != nil` will not understand why both paths exist or when each is taken.

**Fix**: Add a comment explaining the dual path, or remove the in-memory path entirely (see B2). If kept, document: "// When ps.Index is non-nil (immediately after IndexArchives, before next restart), serve from the in-memory snapshot to avoid a DB round-trip. After restart, only the DB path is active."

**Effort**: Trivial.

---

### E4 — `migrate()` migration logic is dense and needs section comments

**File**: `src/internal/archive/storage.go:265-397`

**Finding**: The migration function has three distinct phases: (1) CREATE TABLE IF NOT EXISTS for all tables, (2) backwards-compat ALTER TABLE ADD COLUMN, (3) profile table schema recreation. These phases are not labeled. A new developer reading this function cannot quickly identify what each block does or why.

**Fix**: Add a comment header for each phase:
```go
// Phase 1: create tables (idempotent — CREATE TABLE IF NOT EXISTS)
// Phase 2: backwards-compatible column additions for existing databases
// Phase 3: profile table recreation for multi-user schema upgrade
```

**Effort**: Trivial.

## Part F — Frontend: App.svelte Decomposition (LARGE — SEPARATE SESSION)

**File**: `src/frontend/src/App.svelte` (1369 lines)

**Finding**: The entire application — login, provider selection, gallery, messages, data explorer, settings, modals — lives in a single 1369-line Svelte file. This violates single-responsibility and makes it very hard to reason about state (dozens of variables, reactive statements interleaved with markup). By Zacatl's code-style standard ("files should match their primary responsibility"), this file should be split into components.

**Proposed decomposition**:
```
src/frontend/src/
  App.svelte           (shell, routing, global state)
  components/
    LoginModal.svelte
    ProviderSelector.svelte
    Dashboard.svelte
    Gallery.svelte
    GalleryFilters.svelte
    Messages.svelte
    MessageThread.svelte
    DataExplorer.svelte
    MediaModal.svelte
    SettingsModal.svelte
  lib/
    mediaUrl.ts        (mediaHttpSrc helper)
    messageTypes.ts    (messageTypeLabel helper)
    platformTheme.ts   (accent variable setter)
    types.ts           (re-export of wails-generated types)
```

**Zacatl alignment**: Mirrors the domain-layer separation — each component owns its own state and communicates via props/events, mirroring infrastructure ↔ domain ↔ application separation.

**Note**: This is the largest item in this audit. It is **not a quick fix** — it requires restructuring all reactive state, deciding which state is global vs component-local, and verifying the full UI flow after split. Recommend doing this in a dedicated session after the higher-priority SQL/dead-code items are closed.

**Effort**: High (2–3 sessions minimum).

## Part G — Error Handling Improvements (MEDIUM PRIORITY)

### G1 — `ServeHTTP` returns 404 for both "not found" and "internal server error"

**File**: `src/appshell/app.go:367-370`

```go
data, ext, err := a.service.MediaBytesForUser(platform, userId, id)
if err != nil || data == nil {
    http.NotFound(w, r)
    return
}
```

**Finding**: If `MediaBytesForUser` returns an error (DB error, zip read failure), the client receives a 404 Not Found. This hides infrastructure failures as apparent content gaps. The browser's DevTools or logs will show `GET /media/... 404` with no indication that an error occurred on the server.

**Fix**:
```go
data, ext, err := a.service.MediaBytesForUser(platform, userId, id)
if err != nil {
    http.Error(w, "internal server error", http.StatusInternalServerError)
    return
}
if data == nil {
    http.NotFound(w, r)
    return
}
```

**Effort**: Trivial.

---

### G2 — `GetPlatformStats` silences the UNION ALL query error via `if err == nil`

**File**: `src/internal/archive/service.go:276`

```go
rows, err := s.store.db.Query(...)
if err == nil {  // error ignored — dashboard shows 0/0 silently
```

**Finding**: This guard means a DB-level error on the image/video count query returns a partially-populated stats struct with `ImageCount: 0, VideoCount: 0` and no indication of failure. (This is the same block that also violates the layer boundary in C1.)

**Fix**: Once C1 is resolved (remove the direct `s.store.db` access and trust `Store.PlatformStats`), this issue disappears. Part A5 addresses error propagation inside `Store.PlatformStats` itself.

**Effort**: Resolved by C1 fix.

## Part H — Robustness & Minor Issues (LOW PRIORITY)

### H1 — `platform_snapshots` ON CONFLICT clause will conflict with A1 fix

After fixing A1 (adding `user_id` to the PK), the ON CONFLICT clause:
```sql
ON CONFLICT(platform) DO UPDATE SET ...
```
must become:
```sql
ON CONFLICT(platform, user_id) DO UPDATE SET ...
```

This is tracked here as a dependency of A1, not a separate item.

---

### H2 — `SaveProfile` doesn't use a transaction for the "log in" flow

**File**: `src/internal/archive/storage.go:869-894`

**Finding**: `SaveProfile` does a "set all to logged_out" UPDATE then either INSERT or UPDATE for the target profile. If the process dies between these two statements, all profiles are logged out and the intended login never completed. Using a transaction would make this atomic.

**Fix**: Wrap both operations in a single transaction.

**Effort**: Low.

---

### H3 — `Conversation` method in storage does a full table scan to find one conversation

**File**: `src/internal/archive/storage.go:252`

```go
func (s *Store) Conversation(platform, id string, userId int64) (*types.Conversation, error) {
    convos, err := s.loadConversations(platform, true, userId)
    // ... then linear scan
```

**Finding**: `loadConversations` with `includeMessages=true` scans all conversations and loads all their messages, then the caller linearly searches for one conversation by ID. For a user with many conversations (the test archive has hundreds), this loads thousands of messages just to return one conversation.

**Fix**: Add a `Store.ConversationByID` method that queries `WHERE conversation_id = ?` directly, and separately loads messages only for that one conversation.

**Effort**: Medium — new query + new method + update call site in service.go.
