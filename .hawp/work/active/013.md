# 013: Auto-update support for packaged app

**Status:** in-progress
**Created:** 2026-07-20
**Updated:** 2026-07-21 — slices 1–2 implemented; slice 3 has a full, precise implementation plan (exact code, ready to execute, not yet built)

---

## Problem

The app ships as a Wails-built native binary. There was no version identity, no release channel, and no update mechanism — installed builds silently diverge from the repo. The project is going open source, so releases must be reproducible from CI on GitHub.

## Slices

### ✅ Slice 1 — Version identity (done 2026-07-20)

- `main.go` has `var version = "dev"`, injected at build time via `wails build -ldflags "-X main.version=v0.1.0"`.
- `appshell.App` exposes an `AppVersion()` Wails binding; the Settings modal shows the live version instead of a hardcoded "v1.0.0".
- `wails.json` carries real product info (product name, version 0.1.0, company `sentzunhat`, MPL-2.0 copyright); the macOS bundle identifier is `com.sentzunhat.mochila-archive-viewer` in both Info.plists.
- `build/appicon.png` is a real branded icon (red mochila on the app yellow, macOS rounded-rect) — Wails derives the .icns/.ico from it at build time.

### ✅ Slice 2 — Release pipeline (done 2026-07-20, unverified until first tag)

- `.github/workflows/release.yml`: on `v*` tag push, builds darwin-arm64, darwin-amd64, and windows-amd64 with the tag injected as the version, zips the artifacts, and attaches them to a **draft** GitHub Release with generated notes. Draft so the maintainer reviews before publishing.
- To cut a release: `git tag v0.1.0 && git push origin v0.1.0`, then publish the draft on GitHub.
- **Verify on first tag**: the workflow has not run yet; watch the first run for wails CLI/toolchain issues.

### ▶ Slice 3 — Update check (plan finalized 2026-07-21, ready to execute)

**Full implementation plan:** [update-check-implementation-plan.md](../../kit/examples/update-check-implementation-plan.md) — exact code for `archive.CheckForUpdate`/`isNewer` (Go, dependency-free dotted-version compare), the `App.CheckForUpdate()` binding (deliberately no `error` return — every failure mode already folds into "no update available"), where it plugs into the existing Settings "About" section, and a table-driven test following the `storage_test.go` pattern.

Summary:
1. Startup + manual "Check for updates" button in Settings fetch `https://api.github.com/repos/sentzunhat/mochila-archive-viewer/releases/latest`, 3s timeout.
2. Any failure (offline, no releases yet, rate limited) → silently "no update," never an error the user sees.
3. `dev` builds never nag about updating.
4. No auto-download — just a link to the release page.
5. Backend gains an `AppSettings.UpdateCheckEnabled` field for a future opt-out, but no UI toggle this slice (the existing `AppSettings` struct isn't persisted to disk at all — it's a process-lifetime global — so a toggle wouldn't survive a restart until that's fixed separately; noted as a known limitation, not blocking this slice).

### ⏸ Slice 4 — In-place self-update (deferred)

Gated on code signing + notarization. Replacing a signed .app bundle piecemeal invalidates its signature; needs a full-bundle swap (e.g. `minio/selfupdate` on the inner binary won't survive Gatekeeper). Revisit once an Apple Developer ID exists.

## Risks

- Unsigned release artifacts will be Gatekeeper-quarantined on download (right-click → Open). Document this in the README release section until signing exists.
- GitHub API rate limits are irrelevant at 1 unauthenticated request/launch.
